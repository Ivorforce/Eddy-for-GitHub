# Eddy for GitHub

A local Flask app that turns GitHub's notification firehose into a "you should look at this" view. Single user, runs as `source .venv/bin/activate && python -m app run` and serves on `localhost:5734`. Auth via `gh auth token` (with `GITHUB_TOKEN` env override).

The point is *triage* ‚Äî surface what needs attention, quiet what doesn't ‚Äî not a fancier inbox. AI integration carries most of the noise reduction; the local infrastructure is the substrate that gives it useful signal to work with. v1 is event-sourced ‚Äî see "AI v1" below.

## Stack

Flask + Jinja + HTMX + Pico CSS + SQLite. Static assets vendored under `static/vendor/`; UI icons are Octicons (vendored there, inlined as `<path>` data in templates with `fill="currentColor"` ‚Äî the swirly-eyes brand mark in `static/icon.svg` / the header is ours ‚Äî alternate `icon-a.svg` / `icon-c.svg` kept around). Bundled/depended third-party licenses are reproduced in `THIRD_PARTY_LICENSES.md`. Python 3.11, stdlib `venv` + `pip`, no lockfile. Schema migrates via a hand-rolled version ladder in `app/db.py` ‚Äî add a new `SCHEMA_VN` rather than editing existing ones. User-flippable toggles live in `config/settings.toml` (TOML; defaults + spec in `app/settings.py`, doc surface is `config/settings.example.toml`); the `meta` table is for runtime bookkeeping (`schema_version`, `last_poll_at`, HTTP `last_modified_*`) only.

## Cross-cutting principles

**GitHub is the source of truth.** The local DB is an augmented cache: stores its own state (favorites, notes, baselines) but doesn't fight GitHub on shared state (read/unread, ignored). The app should remain useful as a "fancier inbox" even with no AI active.

**Indicators persist across dismissals; engagement re-anchors them.** Mark-read / done / mute / snooze keep the "+N comments", `+ mentioned`, review-state newness visible so the user can see what they just handled. *Engagement* ‚Äî in-app visit, observed read-on-github, or self-authored comment/review/body_edit ‚Äî re-anchors the since-visit window (the user demonstrably looked). Pill activity counts come from `thread_events` with `ts > last_engaged_at`; the mention indicator is a `mention` thread_event keyed on the notification's `updated_at`. See `github._clear_engagement_baselines`, called from `_apply_action(visited)`, `_upsert`'s `read_on_github` branch, and `_enrich`.

**Combined poll, unread-fetch fallback.** Each auto-poll runs `?all=true&per_page=100`; every item carries its own `unread` flag so `_upsert` reconciles read-state for whatever the response contains. When local has anything unread we add `since=<earliest local unread.updated_at>` (and skip the conditional) so the response narrows to the activity window that could affect local state ‚Äî the feed's Last-Modified tracks activity not `last_read_at`, so a 304 would hide silent reads. With zero local unreads, fall back to a cheap `If-Modified-Since` 304. The dedicated unread fetch (`/notifications` + reconciliation) only fires on `force_full` (manual refresh, app launch) or when a locally-unread row sits outside the latest-100 window ‚Äî see `_fetch_combined` / `_has_unread_outside_window`. Deeper history goes through Backfill.

**Auto-refresh cadence.** Default **Live** (5 min + tab-focus refresh) optimizes for freshness; **Hourly** (xx:00) and **Daily** (04:00 local) are wall-clock-anchored to make the schedule predictable and visible ‚Äî the lever is reducing the compulsive-check pull, not just throttling fetches. **Manual** never auto-polls. Focus-triggered refresh is Live-only ‚Äî in the other modes, refocusing the tab learns nothing fresh so the pull fades. The loop catches up at launch when a tick was missed during downtime. Setting `auto_refresh` (in `config/settings.toml`) + meta key `last_poll_at`; see `poll._is_due` and the status-pill tooltip in `templates/base.html` (`nextTickLabel`).

**Synthetic rows.** GitHub has no notification handle for an arbitrary issue/PR/discussion, so things pulled in outside the poll ‚Äî Backfill ‚ñ∏ "My open issues & PRs" (search API; `github.backfetch_issues`) and Backfill ‚ñ∏ "Track a link" (one pasted github.com URL; `github.track_link`) ‚Äî land as rows with a `q_<node_id>` id and `reason='manual'`, pre-read, thread-mutating actions short-circuiting (`_is_synthetic`). `_upsert` collapses a synthetic row when the real notification for that thread later arrives.

**Archive is soft; Mute is permanent.** Done (`action='done'`, hidden by default) resurfaces ‚Äî `action`‚ÜíNULL ‚Äî the next time a poll re-delivers the notification, which GitHub only does on new activity (mirrors github.com's Done auto-reset). Mute (`action='done'` + `ignored=1`) doesn't: the unsubscribe stops delivery. AI re-judgment never resurfaces a row ‚Äî only GitHub activity (or clicking Done again). Lives in `_upsert`; no marker event for it ‚Äî whatever activity triggered the re-delivery is already in the timeline, and `action`‚ÜíNULL is the state record (a *manual* un-archive logs `undone`).

**Snooze hides with a wake timer.** `action='snoozed'` + `notifications.snooze_until`; archived on GitHub like Done, resurfaced unread by `poll._wake_snoozed` when the timer passes ‚Äî or sooner on new GitHub activity, since plain snooze stays subscribed. The **quiet** variant (popover checkbox ‚Üí `&quiet=1` on `POST /set/<id>/snooze`) *also* unsubscribes (`ignored=1`) for the duration ‚Äî for a busy thread you want a periodic digest of, not a live feed ‚Äî so nothing resurfaces it early; the wake re-subscribes. `ignored` is the only thing that distinguishes the two flavours. See `set_snooze` / `_wake_snoozed`.

**Per-thread kind filter (user-controlled, no AI).** GitHub thread subscriptions are all-or-nothing; `notifications.muted_kinds` lets the user mute individual activity types (`comment` / `review` / `code`; `lifecycle` ‚Äî merge/close/reopen/answered ‚Äî always notifies and isn't mutable, shown as a disabled menu row) on a thread via the ‚ñæ dropdown on the Mute segment. The mutable set and which type produces which live in `github.MUTE_KINDS` / `MUTE_KINDS_BY_TYPE`. A re-delivery whose new activity is *entirely* muted kinds gets **absorbed** ‚Äî `github._apply_mute_filter` re-applies the prior row state, advances baselines past the activity, and freezes `effective_updated_at` (the local sort key) so the row keeps its slot. Directed reasons (`mention` / `team_mention` / `assign` / `review_requested`) override the mute. `muted_kinds` / `baseline_head_oid` / `effective_updated_at` are in the "not touched by `_apply_action`" club. See the `SCHEMA_V25` comment in `app/db.py` and `_apply_mute_filter` for the mechanics.

**Bystander throttle (global, no per-thread UI).** Bursts of `comment` / `code` activity on a thread the user is only watching surface once, then a 30 min window (`notifications.throttle_until`) freezes `effective_updated_at` so the row keeps its slot; when it expires, the next activity (or `_release_throttled`) bumps once with the accumulated "+N new comments" behind it. Exempt: directed reasons, involved reasons (`author` / `comment` / `manual` / `your_activity`), tracked threads, archived/snoozed rows, and any delivery that includes `review` or `lifecycle`. Toggle is the setting `quiet_bystanders` in `config/settings.toml` (default on; `POST /settings/quiet_bystanders`). See `github._apply_throttle`.

## AI v1 (event-sourced)

User-triggered, **advisory** per-thread judgment with **episodic memory**. Each thread carries a chronological event log (`thread_events` table) ‚Äî every comment, review, AI verdict, row-state user action, and free-text user message. The AI sees the full timeline on every judgment, so it reasons about *what's changed since last time* rather than re-classifying a thread from scratch ‚Äî and on a re-judgment a derived `changes_since_last_verdict` roll-up (events after the last `ai_verdict`; see `ai._changes_since_last_verdict`) hands it that diff instead of making it re-scan the flat log.

Lives in the Relevance column ‚Äî two rows, same shape in both modes (toolbar Options ‚ñ∏ "AI triage mode" toggles): a **pill group** on top, the **priority picker** below (`PRIORITY_UI` ‚Üí `POST /set/<id>/priority`, logs a `priority_change` event; unpinned cell falls back to the AI verdict's score, or neutral in manual mode). The pill's left half opens the timeline popover ‚Äî reachable even with no verdict. Both modes share `_recency_summary` (event-counting past an anchor ‚Äî last verdict for AI, `last_engaged_at` for manual) and the structured tooltip (description on top, hairline, `Since this assessment:` / `Since last visit:` / `Since first notification:` + breakdown below); see `_attach_pill_status`. Manual mode left = bright comma-joined activity counts when present; otherwise note (amber) > merge_state > action_needed > pr_review_state > reason fallback (all muted). Icon tints amber when the thread is never-engaged (`never_visited`, parallel to AI's `unassessed`). Right = the note icon ‚Üí per-thread note popover (`POST /note/<id>`, amber when a note exists). AI mode left = the verdict's `description` echo (ellipsized; full text in the tooltip / popover) or `unassessed` ‚Äî the suggested action / track delta aren't spelled out here, only as purple rings on the Actions column ‚Äî right = the assess/re-assess trigger (`POST /ai/<id>/judge`) ‚Äî greyed + disabled when `ai_uptodate` (a verdict exists, no `_VERDICT_INVALIDATING_KINDS` event ‚Äî `comment` / `review` / `lifecycle` / `user_chat` / `body_edit` / `code` ‚Äî since), else purple. An outdated verdict gets a dashed pill border.

Verdicts are advisory only: the pill / priority color shape *display*, but no row state is auto-applied. The user takes their own row actions (visit, mark read, mute, archive, track). Those land as `user_action` events in the timeline; the next judgment compares them against the prior verdict and recalibrates ("I suggested ignore, they visited ‚Üí I underestimated interest").

**Inputs:**

- `config/preferences.md` ‚Äî free-text user prefs (interests, important repos / people, noise patterns). Loaded into the cached system block. `config/preferences.example.md` ships as a template.
- `app/ai_system_prompt.md` ‚Äî shipped instructions (cost asymmetry, brevity rules, output-field semantics, **timeline interpretation**). The "do not restate row-visible facts" rule is load-bearing, as is "the `description` is self-contained ‚Äî written fresh each judgment, never referencing prior verdicts (no 'unchanged'/'as before')"; "user_chat is authoritative for this thread", "user_action after a verdict is calibration feedback", and "prior verdicts are calibration input, not something to cite" are the v1 reading-rules.
- The per-thread context payload (`_load_thread_context`, serialized into the user message). Every ISO timestamp in it is suffixed with a coarse precomputed age ‚Äî `2024-12-16T19:35:06Z (1.4y ago)` ‚Äî by `_annotate_ages`, so the model never subtracts dates by hand (Haiku gets multi-month deltas badly wrong). Anchored full-match only, so timestamps inside comment/note bodies are left alone.

**Verdict shape (one forced tool call ‚Äî `judge_thread` or, on a re-judgment, `skip`):**

```python
judge_thread({
    disposition:          "look" | "queue" | "mute" | "done" | "snooze" | "snooze_quiet",
    snooze_days:          int,                  # with snooze (when to chase) or snooze_quiet (digest interval)
    set_tracked:          "track" | "untrack" | "leave",
    priority_score:       float ‚àà [0, 1],       # bucketed to low/normal/high for color
    subscription_changes: ["mute_<kind>" | "unmute_<kind>", ...],  # usually []; per-thread mute_kinds tweaks
    description:          str,                  # the standing take ‚Äî self-contained, written fresh each time, never citing prior verdicts
    reply:               str,                  # optional; a direct reply to a user_chat ‚Äî omitted when there's nothing to say
})
```

`disposition` is the row's inbox-state outcome ‚Äî `look` (personal engagement required; row stays highlighted), `queue` (triaged, marked read, row stays visible in the priority-sorted act-on-it queue), `done` (the user's part is done and they're fine if nothing happens again; Done auto-resurfaces on the next activity, so it's not a "park it for now"), `snooze` (waiting on something ‚Äî back at ~`snooze_days`, or sooner on activity; a nudge if it stalls), `snooze_quiet` (the thread's a firehose ‚Äî unsubscribe *and* hide for ~`snooze_days`, then a re-subscribed fresh look; no early resurface ‚Äî re-doing it is a periodic digest), `mute` (full opt-out ‚Äî hides the row *and* unsubscribes for good, never resurfaces). Rule of thumb: never `done` where silence would be a problem ‚Äî that's `snooze`'s timer. The "stay visible and subscribed but quiet the churn" middle ground is `subscription_changes`, not `disposition`. The verdict's `snooze_quiet` value maps onto the UI's quiet-snooze flavour (`set_snooze` `&quiet=1`) ‚Äî see "Snooze hides with a wake timer" above. `subscription_changes` is a forward-looking, advisory delta on the thread's `muted_kinds` (which the AI now sees in its input); the per-thread tool schema only offers the kinds that fire on that type, in the right mute/unmute direction. `reply` is the one non-standing field ‚Äî a conversational answer when a `user_chat` asked something; the prompt tells the AI to omit it rather than emit filler. Nothing auto-applies. The verdict cache (`notifications.ai_verdict_*`) is never auto-cleared ‚Äî Re-ask overwrites it (and `_save_verdict` keeps the prior verdict in `thread_events` for the next judgment to see).

**Economical re-judgments.** On a re-judgment a verdict can be partial or absent: `judge_thread` may *omit* `disposition`/`priority_score`/`description` to keep the last verdict's value (the merge in `judge` fills them from `ai_verdict_json` ‚Äî which is always stored complete, so inheritance never leaves a gap), and on a plain Re-ask the model may instead call **`skip`** (no args) to re-affirm the prior verdict wholesale. Whether `skip` is offered (re_evaluate + a prior verdict exists) and whether the standing fields stay in `required` is built per-call (`_build_tool_def`, `SKIP_TOOL_DEF`, `judge`). An inherited `priority_score` ‚Äî and a `skip` ‚Äî won't clear a hand-set `priority_user`; a freshly supplied one will (`_save_verdict(reclaim_priority=)`). A `skip` still lands an `ai_verdict` event (the prior payload) and bumps `ai_verdict_at`, so it clears the outdated-border and advances the timeline's "AI last looked" mark. With thinking off the tool_choice is `{"type":"any"}` (forces a tool, lets the model pick judge-vs-skip ‚Äî that's exactly the boring re-judgment case).

In **AI mode**, the verdict's suggestions show as purple inset rings on the matching Actions-column controls ‚Äî `disposition` ‚Üí the Ignore / Done / Mute / Snooze button, `set_tracked` ‚Üí the Track button, an unapplied `subscription_changes` ‚Üí the ‚ñæ caret (plus the affected option(s) inside its menu, plus an "‚ú¶ apply suggestion" row that bulk-applies via `POST /ai/<id>/apply-mute-suggestion`). Every ring clears once that suggestion is in effect, so the purple only ever means "act on this", never "undo this" or "this is special" (`look` has no button, so it rings nothing). Computed in `_row_to_dict` as `action_pending` / `track_pending` / the `pending_*` lists on the verdict dict; manual mode shows none of it.

**`thread_events` table (SCHEMA_V20):** chronological per-thread log keyed by `(thread_id, kind, external_id)` where present. The unique partial index makes re-fetched comments / reviews idempotent ‚Äî same GitHub `databaseId` UPDATEs the payload (catches body edits) instead of appending. Event kinds:

- `comment` / `review` / `lifecycle` / `body_edit` / `code` ‚Äî GitHub-side activity, source `github`. Written by `_enrich` in `app/github.py`. `body_edit` (issue/PR/discussion description edited) is keyed on the edit timestamp ‚Äî one event per distinct edit; not a `MUTE_KINDS` member, and counts as new context for the AI (it only sees the current body, never a diff). `code` (PR head commit advanced) is keyed on the new abbrev_oid and carries `{prev_oid, committed_at, additions, deletions, changed_files}` so a re-judgment can compare per-push scope against the prior verdict ("small fix" vs +500/-50). First-enrichment is a no-op for both, same as `baseline_comments`.
- `ai_verdict` ‚Äî the AI's verdict, source `ai`. external_id joins back to `ai_calls.id` so the full request / response is one query away. Written by `ai.judge` on success.
- `user_action` ‚Äî `visited` / `read` / `read_on_github` / `done` / `muted` / `undone` / `unmuted` / `kept_unread` / `snoozed` / `unsnoozed` / `woken` / `absorbed`. source is `user` (clicked in our app) or `github` (observed/applied remotely ‚Äî `read_on_github`; `woken` when a snooze timer elapses; `absorbed` when `_apply_mute_filter` re-suppresses a muted-only re-delivery). Written by `_apply_action`, except `read_on_github` (the `_upsert` path in `app/github.py`), `absorbed` (`_apply_mute_filter`), and `woken` (`poll._wake_snoozed`). A Done thread that activity resurfaces gets *no* event ‚Äî the activity itself is the record.
- `user_chat` ‚Äî free-text per-thread message, source `user`. Written by the composer (see Timeline popover) ‚Äî both `POST /ai/<id>/chat` (free Note) and `POST /ai/<id>/judge` with text (Send to AI) record one. NULL external_id; each save is its own event (no dedup).

**Timeline popover (`templates/_row.html` + `_timeline_event.html`):** chronological conversation log, rendered in both modes. The event list is fetched lazily ‚Äî the row ships an empty `<ol>`, HTMX pulls `GET /timeline/<id>` (‚Üí `_timeline_list.html`, built by `_build_timeline`) on each popover-open ‚Äî so the formatted, markdown-rendered timeline stays out of every table swap; only the cheap verdict-derived bits the row itself shows (pill `recency`, `ai_uptodate`) are attached eagerly (`_attach_verdict_status`). AI verdicts and user chat render as full chat bubbles (AI purple-right, user blue-left); in a verdict the standing `description` is always a recessed aside, and a `reply`, when present, sits above it as the body. Adjacent dismissal user_actions are coalesced (`_coalesce_user_actions`), recurring user_actions that repeat every activity round (visit / mark-read / archive) dedupe globally to their latest occurrence (`_coalesce_recurring_actions`, `_RECURRING_ACTIONS`), runs of `body_edit` collapse to the latest (`_coalesce_body_edits`), and only the most recent `ai_verdict` renders ‚Äî earlier ones drop, with a small "‚Ü∫ N" marker on the survivor (`_drop_superseded_verdicts`). A `skip` is naturally subsumed: it lands a fresh `ai_verdict` row with the prior payload, so the survivor ends up being that row. All visual-only ‚Äî the AI still sees every event in `thread_events`. **AI mode only:** a composer at the bottom with two acts ‚Äî Enter saves a free Note (`/ai/<id>/chat`, no AI call), Ctrl+Enter sends to the AI (`/ai/<id>/judge` with the text ‚Äî "‚Üª Re-ask" when empty, "Send to AI" with text). Defaulting Enter to the free Note matches the cost asymmetry (a judgment is ~0.5¬¢ + latency; a note costs nothing and the AI picks it up next time it judges).

**No autonomous re-judgment in v1** ‚Äî explicit Ask AI / Re-ask only. Per-thread `policy` field intentionally absent. Add when auto re-judging ships; the natural trigger is "N new events since latest ai_verdict" exceeding a threshold.

**Storage:** every API call writes a row to `ai_calls` (full request + response, token breakdown, estimated cost, status). Useful for prompt tuning and the soft daily cap (`AI_DAILY_CAP_USD`, default $2). Past verdicts also live as `ai_verdict` events on `thread_events` ‚Äî joined via `external_id`.

**Introspecting one item.** The SQLite DB is `data/notifications.db`. To see exactly what the AI saw and thought for a given thread, look up its `notifications` row by `html_url` (or `id`), then read `ai_calls` for that `thread_id` ‚Äî `request_json` is the assembled prompt, `response_json` has the model's thinking blocks and tool call.

**Default model:** Haiku 4.5. Configurable via `AI_MODEL`. The system prompt + preferences sit well past Haiku's 4096-token cache minimum, so `cache_control` markers on both fire routinely ‚Äî typical cached prefix is ~10-12K tokens; cache hits read at 10√ó cheaper than fresh input. Misses are the 5-min ephemeral TTL expiring between bursts.

**Thinking budget.** Extended thinking (~4k-token budget) on every judgment *except* one case: a Re-ask with a `code` push since the verdict and no `_THINKING_REQUIRED_KINDS` event (`comment` / `review` / `lifecycle` / `user_chat` / `body_edit`) ‚Äî that runs thinking-off with `tool_choice` forced (mutually exclusive on the API), since the fresh file list / diff stats / `last_commit` are already in the prompt. A Re-ask with *nothing* new since the verdict still thinks ‚Äî the pill's rerun is greyed then, so reaching for the popover's "‚Üª Re-ask" signals "I want a deeper take". Thinking is ~40% of a cache-warm call's cost. See `ai._should_think` / `_THINKING_REQUIRED_KINDS`.

**Mode toggle plumbing:** `triage_mode` is a persisted user setting, not a filter ‚Äî stored in `config/settings.toml` (`_get_triage_mode` / `_set_triage_mode` in `app/web.py`, backed by `app/settings.py`). The "AI triage mode" row in the toolbar's Options dropdown posts to `/settings/triage_mode`, which flips the value and returns the re-rendered table; the row flips its own checkmark client-side. Per-row HTMX swaps read the mode server-side, so it doesn't need to ride the request.

**History compression** (deferred): timelines are uncompressed in v1. When threads grow long enough to matter, the hook is `kind=ai_recap` events that summarize older history; the AI reads recaps in lieu of the events they replaced.

## How the user works

- One coherent change per commit. Brief message, *why* over *what*. Co-authored by Claude Opus.
- Pushes back on suggestions; expects pushback in return. "Clean wins" is the recurring frame.
- Boring stable tech preferred. No feature-flag / fallback proliferation.
- Often probes edge cases ("what if I do this on mobile?") before settling UX.
